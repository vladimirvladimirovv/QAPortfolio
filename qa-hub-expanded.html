QA Best Practices

Quality Assurance isn't about finding bugs — it's about preventing them, reducing risk, and building confidence in what we ship. These practices are the foundation of effective QA work.

Core Principles

🎯
Shift-Left Testing

Engage early in the development lifecycle. Review requirements, participate in design discussions, and identify risks before code is written. Bugs found early cost 10x less to fix than bugs found in production.

🔍
Risk-Based Testing

Not all features carry equal risk. Focus testing effort on high-impact areas: payment flows, authentication, data integrity, and user-facing critical paths. Prioritize based on business impact and likelihood of failure.

🔄
Test Independence

Each test should be self-contained and reproducible. Tests shouldn't depend on other tests' execution or external state. This ensures reliable results and easier debugging when failures occur.

📊
Data-Driven Decisions

Use metrics to guide testing strategy: defect density, test coverage, escape rate, and time-to-resolution. Let data reveal patterns and inform where to focus effort.

Testing Mindset

Think Like a User

Users don't read documentation. They click randomly, use unexpected data, and make mistakes. Test the product the way real users interact with it — including frustration scenarios, interruptions, and edge cases.

Think Like a Breaker

Your job is to find the ways the system can fail. What happens with invalid input? What if the network drops mid-transaction? What if a user clicks "submit" 10 times? Adopt an adversarial mindset.

Think Like a Detective

When you find a bug, dig deeper. Is this an isolated issue or a symptom of a larger problem? What other areas might be affected? Root cause analysis prevents recurring defects.

Best Practices Checklist

Before Testing

  • Understand the requirements and acceptance criteria
  • Identify test data requirements
  • Set up the test environment correctly
  • Review related bug history
  • Identify dependencies and integration points

During Testing

  • Cover happy path, edge cases, and negative scenarios
  • Document findings as you go
  • Verify fix for related areas (regression)
  • Use realistic test data
  • Note observations even if not bugs
✓
Golden Rule

"Quality is built in, not tested in." Testing validates that quality exists — it doesn't create it. The best QA work happens before testing begins: in requirements review, design discussions, and early collaboration.

Test Strategy & Coverage

A solid test strategy balances thoroughness with practicality. We can't test everything, so we test the right things at the right time with the right depth. This section is designed to help junior QAs understand core concepts and prepare for interviews.

🎯
Interview Prep Note

Interviewers often ask "What types of testing do you know?" or "Explain the difference between X and Y testing." Use this section to build clear, confident answers with real examples.

Understanding SDLC & STLC

🔄
SDLC — Software Development Life Cycle

The complete process of planning, creating, testing, and deploying software.

  1. Requirements — What should the software do?
  2. Design — How will it be built?
  3. Development — Writing the code
  4. Testing — Verifying it works correctly
  5. Deployment — Releasing to users
  6. Maintenance — Ongoing support & updates
✅
STLC — Software Testing Life Cycle

The QA-specific process within SDLC. Know this for interviews!

  1. Requirement Analysis — Understand what to test
  2. Test Planning — Strategy, resources, timeline
  3. Test Case Design — Write test scenarios
  4. Environment Setup — Prepare test environment
  5. Test Execution — Run tests, log defects
  6. Test Closure — Reports, metrics, lessons learned
💬 Common Interview Question: "Walk me through the STLC" ▼

Sample Answer: "The Software Testing Life Cycle has six phases. First, I analyze requirements to understand what needs testing and identify any gaps. Then I create a test plan covering scope, timeline, and resources. Next, I design test cases for different scenarios. Before execution, I ensure the test environment is ready with correct data. During execution, I run tests and log any defects found. Finally, in test closure, I create reports with metrics like pass rate and defect density, and document lessons learned for future sprints."

Testing Levels (Interview Essential)

Testing happens at different levels of the application. Interviewers frequently ask about these — know the differences!

Level What It Tests Who Does It Example
Unit Testing Individual functions/methods in isolation Developers Test that calculateTax(100) returns 10
Integration Testing How modules work together Developers / QA Test that checkout service correctly calls payment API
System Testing Complete application as a whole QA Team Test entire user registration flow end-to-end
Acceptance Testing Business requirements met QA / Product / Users Verify feature matches what stakeholder requested
💡
Remember: "Unit → Integration → System → Acceptance"

Think of it like building a car: test individual parts (unit), test parts working together (integration), test the whole car (system), then let the customer test drive (acceptance).

Testing Types

These are the different approaches to testing. You'll use most of these in your daily work.

🎯
Functional Testing

Validates features work according to requirements. The bread and butter of QA.

  • Verify acceptance criteria
  • Test business logic
  • Validate user workflows
  • Check data processing
🔄
Regression Testing

Ensures new changes don't break existing functionality. Critical before every release.

  • Automate high-value paths
  • Prioritize recently changed areas
  • Focus on critical business flows
  • Include integration points
💨
Smoke / Sanity Testing

Smoke: Quick checks after deployment — can users log in? Do critical paths work?
Sanity: Focused verification of specific changes.

  • 5-10 critical scenarios
  • Run after every deployment
  • Gate for deeper testing
🔗
End-to-End Testing

Tests complete user journeys across the system. Expensive but valuable for critical paths.

  • User registration → checkout
  • Cross-service workflows
  • Third-party integrations
  • Use sparingly — high maintenance
🔬
Exploratory Testing

Unscripted, creative testing based on experience. Finds bugs that scripted tests miss.

  • Time-boxed sessions (60-90 min)
  • Document findings as you go
  • Focus on new/changed areas
  • Use personas and scenarios
🛡️
Non-Functional Testing

Tests quality attributes beyond functionality: performance, security, accessibility.

  • Load and stress testing
  • Security vulnerability scans
  • Accessibility (WCAG compliance)
  • Usability evaluation

Key Comparisons (Interview Favorites)

Interviewers love asking "What's the difference between X and Y?" Here are the most common ones:

🔥 Smoke Testing vs Sanity Testing ▼
Smoke Testing
  • When: After a new build/deployment
  • Scope: Broad — covers all major areas
  • Purpose: "Does the build work at all?"
  • Depth: Shallow — critical paths only
  • Analogy: Turning on a device to see if it powers up
Sanity Testing
  • When: After a bug fix or minor change
  • Scope: Narrow — specific changed area
  • Purpose: "Does this fix actually work?"
  • Depth: Deeper in the specific area
  • Analogy: Checking if a repaired part works

Memory trick: Smoke = "Is there smoke? Something's broken!" (overall health). Sanity = "Is this change sane/logical?" (specific check).

🔥 Retesting vs Regression Testing ▼
Retesting
  • Purpose: Verify a specific bug is fixed
  • Scope: Only the defect that was reported
  • Uses same test case: Yes — exact same steps
  • Example: Bug #123 said login fails → retest login
Regression Testing
  • Purpose: Ensure fix didn't break other things
  • Scope: Related and surrounding features
  • Uses same test case: No — runs broader suite
  • Example: Login fixed → test profile, settings, logout too

Memory trick: Retesting = "Re-do the exact test". Regression = "Regress to check nothing went backwards".

🔥 Verification vs Validation ▼
Verification
  • Question: "Are we building the product RIGHT?"
  • Focus: Process and documentation
  • Methods: Reviews, walkthroughs, inspections
  • Static: No code execution required
  • Example: Review requirements doc for completeness
Validation
  • Question: "Are we building the RIGHT product?"
  • Focus: Actual product behavior
  • Methods: Testing, execution, user feedback
  • Dynamic: Requires running the software
  • Example: Test if feature meets user needs

Memory trick: Verification = "Did we follow the recipe correctly?" Validation = "Does the cake taste good?"

🔥 Black Box vs White Box vs Gray Box Testing ▼
Aspect Black Box White Box Gray Box
Code Knowledge None required Full access Partial knowledge
Focus Inputs & outputs Internal logic & paths Both behavior & structure
Done By QA Testers Developers QA with some code access
Example Testing login UI Unit testing functions API testing with DB checks
Analogy Testing a vending machine from outside Testing with the machine open Testing with partial diagrams

Interview tip: As a QA, you primarily do black box testing. But knowing how to check databases (DBeaver) and APIs (Postman) makes you a gray box tester — which is valuable!

🔥 Severity vs Priority ▼
Severity
  • Definition: Technical impact of the bug
  • Set by: QA (based on technical assessment)
  • Question: "How bad is this bug technically?"
  • Examples: Crash = Critical, Typo = Low
Priority
  • Definition: Business urgency to fix
  • Set by: Product/Business (based on impact)
  • Question: "How urgently must we fix this?"
  • Examples: Homepage bug = High, Admin-only = Low
⚠️
Classic Interview Example

High Severity, Low Priority: App crashes when user uploads a 10GB file (severe crash, but rare scenario)
Low Severity, High Priority: Company logo is wrong on homepage (just a visual bug, but visible to all users and damages brand)

🔥 Test Case vs Test Scenario vs Test Plan vs Test Suite ▼
Term Definition Example
Test Scenario High-level description of WHAT to test "Verify user can log in"
Test Case Detailed steps of HOW to test "1. Go to login page, 2. Enter email..."
Test Suite Collection of related test cases "Login Module Test Suite" (contains all login tests)
Test Plan Document describing overall test strategy Scope, resources, schedule, risk for entire project

Hierarchy: Test Plan → Test Suites → Test Scenarios → Test Cases

Test Design Techniques (Interview Must-Know)

These techniques help you create effective test cases. Interviewers often ask you to apply these!

📊
Equivalence Partitioning

Divide inputs into groups (partitions) that should behave the same way. Test one value from each partition.

Example: Age field accepts 18-65
• Invalid: 0-17 (test: 10)
• Valid: 18-65 (test: 30)
• Invalid: 66+ (test: 70)
📐
Boundary Value Analysis

Bugs love boundaries! Test at the edges of valid ranges, not just the middle.

Example: Age field accepts 18-65
Test: 17, 18, 19, 64, 65, 66
(just below, at, just above each boundary)
📋
Decision Table Testing

For complex business rules with multiple conditions. Create a table of all condition combinations.

Example: Loan approval
• Good credit + High income = Approve
• Good credit + Low income = Review
• Bad credit + High income = Review
• Bad credit + Low income = Reject
🔀
State Transition Testing

For features with different states. Test valid and invalid transitions between states.

Example: Order status
Draft → Submitted → Approved → Shipped
Test: Can order go from Draft to Shipped directly? (Should fail)
🎲
Error Guessing

Use experience and intuition to guess where bugs might hide. Based on common error patterns.

Common guesses:
• Null/empty inputs
• Special characters (!@#$%)
• Very long strings
• Zero and negative numbers
🔗
Use Case Testing

Test complete user scenarios from start to finish. Covers real-world user journeys.

Example: E-commerce purchase
Browse → Add to cart → Apply coupon → Checkout → Pay → Receive confirmation
✓
Interview Power Combo

When asked "How would you test a login form?", combine techniques: Equivalence Partitioning (valid/invalid emails), Boundary Value (password length limits), Error Guessing (SQL injection, empty fields), and Use Case (complete login flow). This shows depth!

Defect/Bug Life Cycle

Understanding how bugs move through the system is essential. You'll be asked about this!

NEW → ASSIGNED → IN PROGRESS → FIXED → READY FOR QA → VERIFIED → CLOSED ↓ REOPENED (if fix didn't work)
Status Meaning Who's Responsible
New Bug just reported, awaiting triage QA creates → Lead reviews
Assigned Bug assigned to a developer Developer owns it
In Progress Developer actively working on fix Developer
Fixed Developer believes it's fixed Awaiting deployment
Ready for QA Deployed to test environment QA to verify
Verified QA confirmed the fix works QA approves
Reopened Fix didn't work, back to developer QA reopens → Developer
Closed Bug is resolved and verified Done!

Coverage Strategy

Coverage Type Target Focus Areas
Requirements Coverage 100% Every acceptance criterion has at least one test
Happy Path Coverage 100% All expected user workflows validated
Edge Case Coverage 80%+ Boundary conditions, limits, unusual inputs
Negative Scenario Coverage 70%+ Invalid inputs, error handling, failure modes
Integration Coverage 90%+ All external touchpoints and APIs
💡
Coverage Trap

100% code coverage doesn't mean 100% quality. Focus on meaningful coverage — testing the right scenarios, not just hitting lines of code. A well-designed 70% coverage can be more effective than poorly-designed 100%.

Quick Reference: Common Interview Questions

Question Key Points for Your Answer
"What is a Test Case?" A set of conditions and steps to verify a specific functionality works as expected. Includes: ID, title, preconditions, steps, expected result, actual result.
"When do you stop testing?" When: test coverage targets met, all critical bugs fixed, deadline reached, risk is acceptable, or budget exhausted. Never "when all bugs are found" — that's impossible!
"What makes a good bug report?" Clear title, reproducible steps, expected vs actual results, environment details, screenshots/logs. Goal: anyone can reproduce without asking questions.
"How do you prioritize testing?" Risk-based approach: focus on high-impact features, frequently used paths, recently changed code, and historically buggy areas. Business-critical flows first.
"What is regression testing?" Re-running tests to ensure new changes haven't broken existing functionality. Done after bug fixes, new features, or any code changes.
"Explain positive and negative testing" Positive: valid inputs, expected behavior (happy path). Negative: invalid inputs, error handling (edge cases). Both are essential!
"What is UAT?" User Acceptance Testing — final testing by actual users/stakeholders to verify the system meets business requirements before go-live.
"What is a defect/bug?" A deviation from expected behavior. When actual result doesn't match expected result based on requirements.
"What's the difference between QA and QC?" QA (Assurance) = process-oriented, prevention focused. QC (Control) = product-oriented, detection focused. QA prevents defects; QC finds them.
"How do you test without requirements?" Exploratory testing, compare with similar features, check industry standards, test common user flows, ask stakeholders questions, document assumptions.
🎯
Interview Success Tips

1. Use real examples: "In my last project, I used boundary value analysis when testing the age field..."
2. Show your thinking: Explain WHY you'd choose a technique, not just WHAT it is
3. Admit what you don't know: "I haven't used that tool, but I'd approach learning it by..."
4. Ask clarifying questions: Shows analytical thinking — "What type of application is this for?"

API Testing Guidelines

API testing validates the business logic layer independent of UI. It's faster, more stable, and catches issues closer to the source. Every QA engineer should be comfortable testing APIs.

HTTP Methods Reference

Method Purpose Idempotent Request Body Common Use
GET Retrieve data ✓ Yes No Fetch user, list items, search
POST Create resource ✗ No Yes Create user, submit form, upload
PUT Replace resource ✓ Yes Yes Update entire user profile
PATCH Partial update ✗ No Yes Update user email only
DELETE Remove resource ✓ Yes Optional Delete user, remove item

HTTP Status Codes Reference

2xx Success

200 OK — Request succeeded
201 Created — Resource created
204 No Content — Success, no body

3xx Redirection

301 Moved Permanently
302 Found (Temporary redirect)
304 Not Modified (Cached)

4xx Client Error

400 Bad Request — Invalid syntax
401 Unauthorized — Auth required
403 Forbidden — No permission
404 Not Found — Resource missing
409 Conflict — State conflict
422 Unprocessable — Validation failed
429 Too Many Requests — Rate limited

5xx Server Error

500 Internal Server Error
502 Bad Gateway
503 Service Unavailable
504 Gateway Timeout

Positive vs Negative Testing

✓ Positive Testing

Validates that the API works correctly with valid inputs.

  • Valid request returns expected data
  • Correct status codes (200, 201)
  • Response schema matches spec
  • Required fields present
  • Data types are correct
  • Pagination works as expected
✗ Negative Testing

Validates that the API handles invalid inputs gracefully.

  • Missing required fields → 400
  • Invalid data types → 422
  • Unauthorized access → 401/403
  • Non-existent resource → 404
  • Malformed JSON → 400
  • Boundary violations → appropriate error

Sample API Test Structure

POST /api/v1/users Create User
Request Body
{ "email": "john.doe@example.com", "password": "SecurePass123!", "firstName": "John", "lastName": "Doe" }
Expected Response (201 Created)
{ "id": "usr_abc123", "email": "john.doe@example.com", "firstName": "John", "lastName": "Doe", "createdAt": "2024-01-15T10:30:00Z" }
Test Cases
Scenario Input Expected
Valid user creation All fields valid 201, user object returned
Duplicate email Existing email 409, "Email already exists"
Invalid email format email: "notanemail" 422, validation error
Weak password password: "123" 422, password requirements
Missing required field No email field 400, "email is required"
SQL injection attempt email: "'; DROP TABLE--" 422, sanitized/rejected

Security Testing Patterns

🔐 Authentication Testing ▼
  • Valid credentials succeed
  • Invalid credentials return 401
  • Expired tokens are rejected
  • Token refresh works correctly
  • Brute force protection (rate limiting)
  • Session timeout enforcement
  • Logout invalidates session
🛡️ Authorization Testing ▼
  • Users can't access others' data
  • Role-based permissions enforced
  • Privilege escalation prevented
  • Admin endpoints protected
  • IDOR (Insecure Direct Object Reference)
  • Horizontal privilege escalation
💉 Injection Testing ▼
  • SQL injection attempts
  • NoSQL injection patterns
  • XSS in input fields
  • Command injection
  • LDAP injection
  • XML/XXE injection
📊 Data Validation ▼
  • Input length limits enforced
  • Data type validation
  • Special character handling
  • Unicode and encoding attacks
  • File upload restrictions
  • Content-Type validation

API Authentication Types

Understanding authentication is critical for API testing. Know these common methods:

🔑
API Key

Simple key passed in header or query parameter.

X-API-Key: your_api_key_here or ?api_key=your_api_key_here

Test: Invalid key, expired key, missing key, key with wrong permissions

🎫
Bearer Token (JWT)

Token-based auth, common in modern APIs.

Authorization: Bearer eyJhbGciOiJIUzI1...

Test: Expired token, malformed token, token for different user, missing Bearer prefix

👤
Basic Auth

Username:password encoded in Base64.

Authorization: Basic dXNlcm5hbWU6cGFzc3dvcmQ=

Test: Wrong credentials, empty password, special characters, case sensitivity

🔐
OAuth 2.0

Industry standard for authorization delegation.

Authorization: Bearer {access_token} + Refresh token flow

Test: Token refresh, scope validation, revoked tokens, PKCE flow

Common API Testing Mistakes

🚫
Avoid These Pitfalls

1. Only testing happy paths — Always include negative tests and edge cases

2. Ignoring response headers — Headers contain important info (rate limits, security)

3. Hardcoding test data — Use variables and environments for flexibility

4. Not validating schema — Check that response structure matches API spec

5. Skipping auth boundary tests — Verify users can't access others' data

6. Testing only in isolation — Test API chains and dependent workflows

7. Ignoring error messages — Error responses should be helpful, not generic

API Testing Workflow

1
Understand the API
Review documentation, Swagger/OpenAPI spec. Understand endpoints, parameters, auth requirements, and expected responses.
2
Set Up Environment
Configure Postman environment with base URL, auth tokens, and common variables. Set up for dev/staging/prod.
3
Test Positive Scenarios
Verify happy paths work: valid requests return correct data with proper status codes.
4
Test Negative Scenarios
Invalid inputs, missing fields, wrong types, unauthorized access. Verify proper error handling.
5
Test Edge Cases & Boundaries
Empty strings, max lengths, special characters, null values, boundary numbers.
6
Test Security
Auth bypass attempts, injection attacks, IDOR, rate limiting, sensitive data exposure.
7
Verify Data Integrity
Check database after create/update/delete operations. Verify data consistency.

Use Postman's Collection Runner to chain requests: Create user → Get user ID from response → Use ID to update user → Delete user. This tests the full CRUD lifecycle automatically.

4. Enter Amex card details

Severity vs Priority

Severity = Technical impact. Priority = Business urgency. They're related but not the same.

Severity Levels

S1 - Critical
System crash, data loss, security breach, complete feature failure
S2 - High
Major feature broken, significant impact, no workaround
S3 - Medium
Feature partially working, workaround exists
S4 - Low
Minor issues, cosmetic defects, typos

Priority Levels

P1 - Urgent
Fix immediately, blocks release or critical business function
P2 - High
Fix in current sprint, significant business impact
P3 - Medium
Fix when possible, plan for upcoming sprint
P4 - Low
Nice to fix, backlog candidate
ℹ️
Example: High Severity, Low Priority

A crash in an admin-only feature used once a quarter is S1 (Critical severity) but might be P3 (Medium priority) if it's not blocking any current work. Conversely, a minor UI bug (S4) on the homepage during a marketing campaign might be P1 (Urgent).

Bug Reporting Anti-Patterns

❌ Bad Bug Reports
  • "It doesn't work" — What doesn't work?
  • "Login is broken" — How? When? For whom?
  • No reproduction steps — Can't fix what can't be reproduced
  • Missing environment — "Works on my machine"
  • No attachments — Screenshots are worth 1000 words
  • Multiple bugs in one ticket — Track separately
✓ Good Bug Reports
  • Clear, specific title
  • Step-by-step reproduction
  • Expected vs actual clearly stated
  • Environment details included
  • Screenshots/recordings attached
  • Investigation notes when available

Investigation Before Reporting

When possible, do preliminary investigation before reporting. This speeds up resolution significantly.

1
Reproduce consistently
Can you make it happen every time? What's the minimum steps?
2
Check browser console
Any JavaScript errors? Failed network requests? Include these in the report.
3
Verify across browsers/devices
Is it browser-specific? Device-specific? This narrows down the cause.
4
Check logs (Grafana)
Look for errors around the timestamp. Include correlation IDs if available.
5
Verify data state (DBeaver)
Is the data correct in the database? Include relevant queries.

AI Tools for QA

AI assistants like Claude can significantly boost QA productivity when used effectively. This section covers practical prompts, use cases, and best practices for integrating AI into your QA workflow.

🤖
AI is a Tool, Not a Replacement

AI excels at repetitive tasks, generating ideas, and processing information quickly. But it doesn't replace QA judgment, domain knowledge, or critical thinking. Use AI to augment your skills, not replace your brain.

High-Value QA Use Cases for AI

📝

Generate comprehensive test scenarios from requirements, user stories, or feature descriptions. AI can quickly identify edge cases you might miss.

🐛
Bug Report Enhancement

Improve bug report clarity, suggest missing details, and help structure reproduction steps for better developer understanding.

🔌
API Test Script Writing

Generate Postman collections, test scripts, and assertions. Convert API documentation into executable test cases.

📊
Test Data Generation

Create realistic test data sets, boundary values, and edge case inputs. Generate data that covers various scenarios.

📚
Documentation Writing

Draft test plans, create user guides, write release notes, and document test strategies faster.

🔍
Code Review Assistance

Analyze code changes for potential bugs, suggest test scenarios based on modified code, identify risky changes.

Effective Prompting for QA Tasks

The quality of AI output depends heavily on your prompt. Here's how to write prompts that get useful results:

1. Be Specific About Context

Include relevant details: what type of application, what feature, what's already tested, what technology stack. More context = better output.

2. Define the Output Format

Tell AI exactly what format you need: table, numbered list, Gherkin syntax, JSON, etc. This saves reformatting time.

3. Specify the Testing Perspective

Mention what types of testing you need: functional, security, performance, accessibility. Ask for specific techniques like boundary analysis.

4. Request Prioritization

Ask AI to prioritize test cases by risk, importance, or execution order. This helps when time is limited.

Best Practices for AI-Assisted QA

✓ Do
  • Always review and validate AI output
  • Use AI for brainstorming and first drafts
  • Provide specific context for better results
  • Iterate on prompts to improve output
  • Combine AI suggestions with domain expertise
  • Use AI to learn new testing techniques
  • Save effective prompts for reuse
✗ Don't
  • Blindly copy-paste AI output without review
  • Share sensitive/proprietary data with AI
  • Assume AI understands your specific context
  • Skip testing because AI "validated" it
  • Rely on AI for final quality decisions
  • Use AI-generated code without testing it
  • Forget to apply critical thinking
⚠️
Security & Confidentiality

Never paste sensitive data into AI tools: customer PII, credentials, API keys, proprietary business logic, or security vulnerabilities. Use sanitized examples and placeholder data. Check your company's AI usage policy before using external AI tools with work data.

Prompt Engineering Tips for QA

Prompt Structure Template Role: "You are a senior QA engineer with expertise in [DOMAIN]" Context: "I'm testing a [TYPE] application that [DESCRIPTION]. The feature I'm testing [FEATURE DETAILS]." Task: "Generate/Create/Analyze/Review [SPECIFIC REQUEST]" Constraints: "Include [REQUIREMENTS]. Exclude [EXCLUSIONS]. Focus on [PRIORITY AREAS]." Format: "Present the output as [TABLE/LIST/JSON/GHERKIN] with columns/fields for [STRUCTURE]" Example: "Here's an example of what I'm looking for: [EXAMPLE]"
💡
Pro Tip: Iterative Refinement

Don't expect perfect results on the first try. Start with a basic prompt, review the output, then refine: "Good start, but also include security test cases" or "Make the steps more specific" or "Add priority levels to each test case." Treat it like a conversation, not a one-shot request.
